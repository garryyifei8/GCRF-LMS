## 🚀 三种部署方式

### 方式一: 一键启动(最快速,推荐⭐)

适用于有网络连接的环境,自动导入镜像并启动。

```bash
# 1. 进入部署包目录
cd 客户演示部署包

# 2. 运行一键启动脚本
./部署脚本/quick-start.sh

# 3. 按照提示操作
# - 选择部署方式(导入镜像 或 Docker Compose构建)
# - 等待服务启动(约1-2分钟)
# - 访问 http://localhost:3011
```

### 方式二: 导入镜像部署(离线环境推荐)

适用于无网络或网络受限的离线环境。

```bash
# 1. 进入部署包目录
cd 客户演示部署包

# 2. 运行导入脚本
./部署脚本/import-image.sh

# 3. 选择镜像文件
# 脚本会列出所有可用镜像,选择最新的即可

# 4. 确认导入并启动
# 按照提示确认,自动完成导入和启动
```

### 方式三: Docker Compose部署(灵活配置)

适用于需要自定义配置的场景。

```bash
# 1. 进入部署包目录
cd 客户演示部署包

# 2. 首先导入镜像
gunzip -c gcrf-web-admin-latest-20251020_011420.tar.gz | docker load

# 3. 使用Docker Compose启动
docker-compose up -d

# 4. 查看状态
docker-compose ps

# 5. 查看日志
docker-compose logs -f
```

## 🔍 常见问题与解决方案

### Q1: Docker未安装或未启动

**现象**: 运行脚本时提示"Docker未安装"或"Docker服务未运行"

**解决方案**:
```bash
# Windows/macOS
# 1. 下载并安装Docker Desktop
# https://www.docker.com/products/docker-desktop

# 2. 启动Docker Desktop
# 双击桌面图标或从应用程序菜单启动

# 3. 等待Docker启动完成(状态栏图标变绿)

# Linux
# 1. 安装Docker
curl -fsSL https://get.docker.com | sh

# 2. 启动Docker服务
sudo systemctl start docker

# 3. 设置开机自启
sudo systemctl enable docker

# 4. 添加当前用户到docker组(可选)
sudo usermod -aG docker $USER
# 需要重新登录生效
```

### Q2: 端口3011被占用

**现象**: 启动失败,提示端口3011已被占用

**解决方案**:

**方法一: 更换端口**
```bash
# 编辑docker-compose.yml
ports:
  - "8080:80"  # 将3011改为8080或其他可用端口

# 或使用docker run命令时指定不同端口
docker run -d -p 8080:80 --name gcrf-web-admin gcrf-library-web-admin:latest
```

**方法二: 查找并释放端口**
```bash
# macOS/Linux
lsof -i :3011
# 找到进程ID(PID),然后kill
kill -9 <PID>

# Windows
netstat -ano | findstr :3011
# 找到进程ID,然后:
taskkill /PID <PID> /F
```

### Q3: 镜像导入失败

**现象**: 导入镜像时报错或超时

**解决方案**:
```bash
# 1. 检查磁盘空间
df -h  # Linux/macOS
dir C:\  # Windows

# 2. 清理Docker缓存
docker system prune -a

# 3. 检查镜像文件完整性
md5sum gcrf-web-admin-latest-*.tar.gz  # 对比meta.txt中的MD5值

# 4. 如果文件损坏,重新获取镜像文件

# 5. 手动导入
gunzip -c gcrf-web-admin-latest-20251020_011420.tar.gz | docker load
```

### Q4: 容器启动后无法访问

**现象**: 容器运行正常,但浏览器无法访问 http://localhost:3011

**解决方案**:
```bash
# 1. 检查容器状态
docker ps | grep gcrf-web-admin
# 应该看到状态为Up

# 2. 检查健康状态
docker inspect --format='{{.State.Health.Status}}' gcrf-web-admin
# 应该返回healthy

# 3. 测试健康检查端点
curl http://localhost:3011/health
# 应该返回 "healthy"

# 4. 查看容器日志
docker logs gcrf-web-admin
# 检查是否有错误信息

# 5. 检查防火墙设置
# Windows: 控制面板 > 系统和安全 > Windows Defender防火墙
# macOS: 系统偏好设置 > 安全性与隐私 > 防火墙
# Linux: sudo ufw status

# 6. 进入容器检查
docker exec -it gcrf-web-admin sh
# 在容器内测试nginx
wget http://localhost/health
```

### Q5: 浏览器显示空白页面

**现象**: 可以访问系统,但页面显示空白

**解决方案**:
```bash
# 1. 清除浏览器缓存
# Chrome: Ctrl+Shift+Delete (Windows/Linux) 或 Cmd+Shift+Delete (macOS)
# 选择"缓存的图片和文件",清除

# 2. 使用无痕模式访问
# Chrome: Ctrl+Shift+N (Windows/Linux) 或 Cmd+Shift+N (macOS)

# 3. 检查浏览器控制台
# F12打开开发者工具 > Console标签
# 查看是否有JavaScript错误

# 4. 检查静态文件
docker exec gcrf-web-admin ls -la /usr/share/nginx/html/
# 应该看到index.html和assets目录

# 5. 检查nginx配置
docker exec gcrf-web-admin cat /etc/nginx/conf.d/default.conf

# 6. 重启容器
docker restart gcrf-web-admin
```

### Q6: 登录失败或无法登录

**现象**: 输入正确的用户名密码,但无法登录

**解决方案**:
```bash
# 1. 确认使用正确的测试账号
# admin / admin123
# librarian / lib123
# operator / op123

# 2. 检查Mock服务是否启动
# 打开浏览器控制台(F12) > Console
# 应该看到: [MSW] Mock Service Worker started

# 3. 检查网络请求
# F12 > Network标签 > 勾选"Preserve log"
# 点击登录,查看API请求是否成功

# 4. 检查mockServiceWorker.js文件
docker exec gcrf-web-admin ls -la /usr/share/nginx/html/mockServiceWorker.js

# 5. 清除浏览器的Application数据
# F12 > Application > Clear storage > Clear site data
```

### Q7: Mock数据修改后不生效

**现象**: 想要修改演示数据,但修改后不生效

**解决方案**:
```bash
# Mock数据存储在内存中,容器重启会重置

# 如需修改数据:
# 1. 修改源码中的mock数据文件
# src/mock/data/books.js
# src/mock/data/readers.js
# src/mock/data/circulation.js

# 2. 重新构建镜像
cd web-admin
npm run build

# 3. 重新构建Docker镜像
docker build -t gcrf-library-web-admin:latest .

# 4. 重启容器
docker rm -f gcrf-web-admin
docker run -d -p 3011:80 --name gcrf-web-admin gcrf-library-web-admin:latest
```

### Q8: 如何切换到真实后端API

**现象**: 需要连接真实的后端服务

**解决方案**:
```bash
# 1. 停止Mock容器
docker stop gcrf-web-admin

# 2. 修改环境变量(需要重新构建)
# 在.env.production文件中:
VITE_USE_MOCK=false
VITE_API_BASE_URL=http://your-backend-server:8080

# 3. 或使用nginx反向代理
# 修改docker-compose.yml,添加环境变量:
environment:
  - BACKEND_URL=http://your-backend-server:8080

# 4. 重新构建并启动
docker-compose build
docker-compose up -d
```

### Q9: 性能问题或响应缓慢

**现象**: 系统运行缓慢,页面加载时间长

**解决方案**:
```bash
# 1. 检查系统资源
docker stats gcrf-web-admin
# 查看CPU和内存使用情况

# 2. 增加Docker资源限制
# Docker Desktop > Settings > Resources
# 调整CPU和内存分配

# 3. 清理Docker系统
docker system prune -a

# 4. 重启Docker Desktop

# 5. 检查网络连接
ping localhost
curl http://localhost:3011/health

# 6. 查看容器日志
docker logs gcrf-web-admin
```

### Q10: Windows系统路径问题

**现象**: Windows系统下脚本无法执行

**解决方案**:
```bash
# 1. 使用Git Bash执行脚本
# 下载Git for Windows: https://git-scm.com/download/win
# 右键选择 "Git Bash Here"
./部署脚本/quick-start.sh

# 2. 或使用WSL2(Windows Subsystem for Linux)
# 在PowerShell中启用WSL:
wsl --install

# 3. 或使用PowerShell执行
# 需要先转换脚本格式:
dos2unix quick-start.sh  # 如果有dos2unix工具

# 4. 或手动执行Docker命令
docker load -i gcrf-web-admin-latest-*.tar.gz
docker run -d -p 3011:80 --name gcrf-web-admin gcrf-library-web-admin:latest
```
